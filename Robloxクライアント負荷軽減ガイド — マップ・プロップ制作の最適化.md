本ドキュメントは、Robloxゲームのマップおよびプロップ制作においてクライアント側の負荷を最小化するための手法を、公式ドキュメント・DevForum・エンジニアの知見に基づいて事実ベースで整理したものである。スクリプティングのパターン・実装手法はスコープ外とする。

ドキュメント末尾には作業チェックリスト（セクション11）・逆引きリファレンス（セクション12）・参考情報源（セクション13）をまとめている。

# 1. 負荷の種類

クライアント側の負荷は、大きく以下の5つに分類できる。各項目の詳しい確認方法・目標値・対策は後続セクションで解説する。

**描画負荷（GPU / Rendering）** — ドローコール数、シーン内のトライアングル数、シャドウ描画、透明オブジェクトの重なりなど。フレームレートに直結する。

**CPU負荷** — 物理演算（Physics）、スクリプト実行、インスタンスの管理・更新処理など。描画のボトルネックがCPU側になるケースも多い。

**メモリ負荷** — テクスチャ、メッシュデータ、コリジョンデータ、インスタンスツリーなどがRAMを消費。2GBデバイス（古いスマートフォン等）ではクラッシュの原因になる。

**ネットワーク転送負荷** — サーバーからクライアントへのレプリケーション。移動する物理オブジェクトやRemoteEventのデータ量。

**ロード時間** — 初回接続時のアセットダウンロードとインスタンス構築。ユーザー離脱率に直結する。

# 2. 描画負荷の最適化

## 確認方法と目標値

**計測前提:** Graphics Mode を Manual・最大値に設定した状態で計測する（セクション10-1参照）。

**何を見るか:** Rendering Stats画面（スタジオ：Shift+F2、クライアント：Shift+F2）の **Draw(Scene)** 欄。ここにドローコール数とトライアングル数が表示される。ゲーム内を歩き回りながら、数値が跳ね上がるホットスポットを探す。

**CPU/GPUどちらがボトルネックか:** 同じ画面に表示される **CPU ms** と **GPU ms** を比較する。高い方がボトルネック。CPU > GPUならスクリプトや物理が原因、GPU > CPUなら描画量が原因。

**目標値の考え方:**

公式ドキュメント（Design for Performance）は固定の正解値を示さず、「ベースライン端末でテストし、フレームレートが落ち始めたエリアの数値をそのプロジェクト固有の上限とする」アプローチを推奨している。落ちたと感じた時点の `Draw(Scene)` の値が、そのゲームの管理上限になる。

**目安値（MrChickenRocket推奨、RDC 2024講演）:**

出発点として、以下の値が講演で示された目安。主にモバイル対象・オープンワールド系ゲームでの実績値であり、カメラ距離・ジャンル・対象プラットフォームによって適切な値は変わる。

- シーン内トライアングル: 500,000以下
- シーン内ドローコール: 500以下
- ほとんどの（90%以上の）スマートフォン・タブレットで品質設定10のまま60FPSを維持できる水準

RDC 2024のスライド（BlackoutCedar記事より引用）では、デバイスごとのトライアングル予算が以下のように示されている。

- ローエンドモバイル: ～100K〜200K tri
- ミドルレンジモバイル: ～300K〜500K tri
- PC/コンソール: 500K〜1M+ tri

## 2-1. ドローコールとインスタンシング（バッチング）

**ドローコールとは**、エンジンがGPUに対して「このオブジェクトを描画せよ」と送る命令のセットです（公式ドキュメントの定義）。1つのオブジェクトにつき最低1回発生し、ドローコールごとにCPUがGPUの描画状態をセットアップする処理が走る。この呼び出し自体のオーバーヘッドが大きく、**ドローコール数が多いほど1フレームあたりのCPU処理時間が増加する**。

ドローコール数とトライアングル数は独立した指標である。「ドローコールが少ないがトライアングルが多い」ケースも「その逆」もあり得るため、どちらか一方だけを最適化しても片方がボトルネックになる。Rendering Stats では両方を合わせて確認する。

ドローコールの削減には2つの独立したテーマがある。**前半はパート自体のインスタンシング**、**後半はDecal/Textureインスタンスの描画コスト**である。混同すると対策が噛み合わなくなるため、分けて理解する。

### パートのインスタンシング

Robloxエンジンは、同じジオメトリかつ同じ外観を持つパートを自動的に1つのドローコールにまとめる「インスタンシング」を行う。これはMeshPart、CSG（Union）、基本Partのすべてに対応している。

外観の一致条件はオブジェクト種別によって異なる。公式ドキュメント（Improve Performance）にはMeshPartについて以下の優先順位が明記されており、Unionと基本PartはSurfaceAppearanceも`TextureId`も持てないため、条件はシンプルになる。

**バッチングが成立する条件（オブジェクト種別ごと）:**

| オブジェクト種別 | ジオメトリ条件 | 外観条件（優先順位順） |
| --- | --- | --- |
| **MeshPart** | 同じ MeshContent（MeshId） | ① SurfaceAppearance が存在する場合 → SA が同一<br>② SA なし・`TextureId` が存在する場合 → `TextureId` が同一<br>③ SA も `TextureId` もない場合 → Material が同一 |
| **基本 Part**（Part / WedgePart 等） | 同じプリミティブ形状 | Material が同一（`TextureId`・SA は使用不可のため） |
| **PartOperation（Union）** | 同じ生成ジオメトリ | Material が同一（同上） |

**バッチを壊さない（異なっていてもOKな）プロパティ:**

- CFrame（位置/回転）
- Size
- Color / BrickColor
- Reflectance
- UsePartColor
- SpecialMeshのOffset、Scale、VertexColor

**バッチを壊す「Batch Wrecker」:**

- 異なるMeshType、MeshId、TextureId
- 異なるSurfaceAppearance（MeshPartのみ）— `ColorMapContent`・`NormalMapContent`・`MetalnessMapContent`・`RoughnessMapContent` のいずれかが異なると不成立
- 異なるMaterial
- Stud構成の違い（面ごとのStud種類の組み合わせで約2,700パターン存在）— かつての LEGO 的なブロッキー表現の名残。現代の制作では全面 `Smooth`（デフォルト）のため通常は問題にならないが、古いアセットの流用時に混入することがある
- **Transparency（半透明）** — 最も影響が大きい。半透明オブジェクトはバッチングが効かず、1オブジェクト＝1ドローコールになる

**実践的なポイント:**

- 同じメッシュを色・サイズ・回転を変えて再利用する「キットバッシング」が非常に有効。MrChickenRocketの例では、複数の木（各4メッシュ構成）が色・回転・スケール違いで配置されていても、わずか2ドローコールで描画されている
- 使用するマテリアルの種類を少なく保つことでバッチ効率が上がる
- 半透明オブジェクトはバッチングが効かず、Overdraw によるGPU負荷も発生する（詳細はセクション2-3）
- `Lighting.Outlines` は無効にする（有効だとPartのインスタンシングが阻害される）
- Unionはユニオン操作のたびにジオメトリが固有生成されるため、見た目が同一でもジオメトリ条件を満たせずバッチが成立しないケースがほとんど。形状の選択肢は以下の3つで、Unionのまま最終成果物にするのが最もバッチング効率が低い：
  - **基本Partのまま** — Block / Wedge / Cylinder 等で表現できる場合は Union 操作自体不要。同形状・同マテリアルで自動バッチングされる
  - **MeshPart化（Blender経由）** — 基本Partで表現できない複雑形状かつ複数箇所に再利用する場合。同一 MeshId を持つ MeshPart 同士はバッチングされる
  - **Union のまま** — 最終成果物として使うのは非推奨（固有ジオメトリによるバッチング不成立・メモリオーバーヘッド・破損リスク）

### Decal / Texture インスタンスのドローコスト

公式ドキュメント（Improve Performance）に明記されている通り、**Decal・Texture・ParticleEmitter はバッチしにくく、存在するだけで親パートとは独立した追加のドローコールが発生する**。パートのインスタンシングとは別の描画コストとして管理する必要がある。

**バッチングが成立する条件:**

- カラーマップのみ使用時: `ColorMapContent` が同一
- PBR マップを使用する場合: `ColorMapContent`・`NormalMapContent`・`MetalnessMapContent`・`RoughnessMapContent` がすべて同一（推定）

（同一コンテンツIDを持つもの同士は一定程度バッチされうる。ただしバッチ効率はパートインスタンシングより低い）

**バッチを壊さない（異なっていてもOKな）プロパティ:**

- Color3（色合い）
- StudsPerTileU / StudsPerTileV（Texture のみ）
- OffsetStudsU / OffsetStudsV（Texture のみ）
- Face（適用する面）

**バッチを壊す「Batch Wrecker」:**

- `ColorMapContent` が異なる — バッチ不成立
- PBR マップ使用時に `ColorMapContent`・`NormalMapContent`・`MetalnessMapContent`・`RoughnessMapContent` のいずれかが異なる — バッチ不成立（推定）
- **Transparency（半透明）** — パートと同様、半透明のDecal/Textureはバッチ不成立

**実践的なポイント:**

- Decal / Texture の使用数を最小限にする
- 同じ画像IDを使い回す（異なるIDを乱立させない）
- MeshPartにSurfaceAppearance（ColorMap）を使う方がパートバッチングとの相性がよく、Decalを別途追加する必要がなくなる

### パッケージの利用

**Package（パッケージ）** とは、MeshPart・Decal・Texture・SurfaceAppearance などを含むモデルをひとまとまりのアセットとして管理する仕組みである。パフォーマンスの観点では直接的な描画削減機能ではなく、**前半・後半のインスタンシング成立条件（同一 asset ID）を確実に維持するための管理ツール**として機能する。

**なぜインスタンシングに関係するか：**
- マップを外部ツールから一括インポートすると、見た目が同一のメッシュでも別々の asset ID が割り振られ、エンジンが「別アセット」と判断してインスタンシングが崩れる
- Package を使うことで、同一メッシュが同じ MeshContent ID を持つことが保証される（パートのインスタンシング成立）

**Decal / Texture への効果：**
- Package には BasePart だけでなく Decal・Texture・SurfaceAppearance も含められる
- Package で管理すれば、Decal の画像 asset ID も複数箇所で確実に統一され、バッチ成立条件（同一コンテンツ ID）を満たしやすくなる

**実践的なポイント：**
- 同じアセットを複数箇所に配置する場合は Package を使って管理する

### FastCluster の注意点

スクリプトで Part のサイズ・スケール・位置を毎フレーム変更すると、エンジンの内部静的バッチ（FastCluster）が毎フレーム再構築されドローコストが増大する。MicroProfiler で `updateInvalidatedFastClusters` が 4ms 以上を占める場合はこの再構築が過剰に発生しているサイン。動くオブジェクトの数と更新頻度を見直す。

## 2-2. トライアングル数の管理

Robloxのメッシュインポート上限は、1メッシュあたり **20,000 tri** である（3Dインポーター・アセットマネージャーともに共通）。

- ワイヤーフレーム表示でエッジが密集して面の空白がほとんど見えない状態は、不必要なポリゴンが多すぎるサインである
- Blenderで内部面（Interior Faces）を削除する。Select → Select all by Trait → Interior Faces → Delete
- シルエットを保ちつつループカットを減らすことで、見た目をほぼ変えずにtri数を大幅削減できる

## 2-3. 透明オブジェクト（Overdraw）

半透明オブジェクト（`Transparency` が 0 でも 1 でもない値）はバッチングが効かず、1オブジェクト＝1ドローコールになる。さらに半透明オブジェクトが重なると、GPU が奥のレイヤーまで何度もシェーディング計算を行う **Overdraw** が発生し、GPU ms が急増する。

**原則:**
- `Transparency` は `0`（不透明）か `1`（不可視）のみ使用する
- 葉・草などのカットアウト表現は `SurfaceAppearance.ColorMap` のアルファチャンネルで形を抜く（メッシュ自体は `Transparency = 0` のまま）
- 半透明オブジェクトを重ねる必要がある場合（窓ガラスなど）は枚数を最小限に抑える

**Overdraw の確認方法:**
Rendering Stats（Shift+F2）の GPU ms が CPU ms を大きく上回る場合、Overdraw が原因である可能性が高い。カメラを透明オブジェクトに向けた状態で GPU ms が跳ね上がるなら確定。

## 2-4. VFX（ParticleEmitter / Beam / Trail）

ParticleEmitter・Beam・Trail はバッチングが効かず、1エフェクト＝1ドローコール以上発生する。また半透明テクスチャを使用するため、大量配置・高設定では Overdraw によるGPU負荷が急増しやすい。装飾目的での多用は避け、マップに配置する環境エフェクト（滝・煙・炎・魔法陣など）は以下の点に注意する。

**ParticleEmitter:**
- `Rate`（毎秒の生成数）と `Lifetime`（粒子の生存時間）の積がシーン内の同時存在パーティクル数を決める。`Rate × Lifetime` を低く抑えることが最も効果的
- 推奨目安: 1エフェクトあたり同時存在パーティクル数を **50〜100以下** に抑える
- `LightEmission`（発光）が高いと描画コストが増大する。発光が不要な煙・霧には `0` を設定する
- Overdrawのコストはパーティクルのスクリーン上のピクセル占有面積と重なる層の数によって決まる。テクスチャの形状（円形か否かなど）はコストに影響しない

**Beam:**
- 数自体を最小限に抑えることが最も効果的

**Trail:**
- `Lifetime` を短く設定するほど画面上に残る軌跡の長さが減り、Overdraw が減少する
- 大量の移動オブジェクトに Trail を付けると急激に負荷が増大するため、必要最小限のオブジェクトにのみ適用する

## 2-5. UI（BillboardGui / SurfaceGui）

看板・モニターなどプロップとして配置する 3D GUI は描画負荷とメモリに影響する。

- `AlwaysOnTop = true` の BillboardGui / SurfaceGui は深度テスト（Zバッファ）を無視するため、オクルージョンカリングが機能せず描画をスキップできない。プロップとして大量配置すると GPU 負荷が増大するため、必要な箇所のみに限定する
- `CanvasSize` を不必要に大きくするとテクスチャメモリを消費する。表示内容に合わせた必要最小限のサイズに設定する

---

## 2-6. RenderFidelity（LOD）

**LOD（Level of Detail）** とは、カメラからの距離に応じてオブジェクトのポリゴン密度を自動的に切り替える技術である。近くでは高精細なメッシュを、遠くでは粗いメッシュを使うことで、見た目の品質をほぼ保ちながらトライアングル数を削減できる。

MeshPartのRenderFidelityプロパティで、距離に応じたLOD挙動を制御できる。デフォルトは **Automatic** である。

**Performance** — 距離に応じてLODを下げる。95%のオブジェクトはこれで良い

**Automatic** — エンジンが自動判定

**Precise** — 常に最高LODを維持。遠景の山など、どの距離からでも形状が重要なものにのみ使用

## 2-7. ライティングモードの選択

`Lighting.LightingStyle` プロパティで選択するライティングモードはパフォーマンスに直結する。

**Soft** — フラットでレトロな見た目。ローカルライトのシャドウ計算がなく最も軽量。大規模・メモリの重いゲーム向き。

**Realistic** — 高品質なシャドウ・ハイライト・反射。新規ゲームのデフォルト設定。追加のシャドウキャスティングライト（PointLight・SpotLight・SurfaceLight）ごとに負荷が増大するため、ライト数の管理が重要。

## 2-8. シャドウの最適化

- `Lighting.LightingStyle` を `Realistic` に設定している場合、シャドウキャスティングライトごとに追加のレンダリングパスが発生し、ドローコールとCPU/GPU負荷が増大する
- 影が不要なオブジェクトは `Part.CastShadow = false` を設定
- ライトの照射範囲（Range・Angle）をできるだけ絞る
- Realisticでのシャドウキャスティングライトの追加は慎重に（Softなら太陽光のみなので比較的軽い）
- 動くジオメトリ（揺れる木など）にシャドウキャスティングライトを当てると毎フレームシャドウ再計算が発生するため避ける

## 2-9. 視認性管理とカリング

### オクルージョンカリング

Robloxエンジンには**自動オクルージョンカリング**が実装されている。壁や地形に隠れたオブジェクトをエンジンが自動検出し、描画をスキップする。開発者側の設定は不要。

ただし以下のケースではエンジンカリングが効きにくいため、手動補完が依然有効：
- **半透明オブジェクト** — 透明なものは遮蔽物として機能しないためカリングされにくい
- **パーティクル・エフェクト** — 通常カリング対象外
- **細粒度の距離制御が必要な場合** — 草・小石など小さなディテールは距離ベースで手動制御する方が確実

**手動オクルージョンカリング（Precalculated Visibility）:** MrChickenRocketの手法として、マップをゾーンに分割し、プレイヤーの位置に応じて見えないゾーンの Model をスクリプトで非表示にする方法がある。

## 2-10. DoubleSided の使用制限

`MeshPart.DoubleSided` を有効にすると、ポリゴンの裏面も描画される。通常 GPU はカメラから見て裏側を向いているトライアングルを描画前に除外する（バックフェースカリング）が、`DoubleSided` はこれを無効化するため、**実効的な描画トライアングル数が最大2倍**になる。シャドウの描画コストも同様に増大する。不要なオブジェクトで有効になっていないか確認する。

# 3. Part / Union / MeshPart の比較

## 3-1. 基本Part

- エンジンに最適化されたジオメトリ。内部的にはメッシュと同様にインスタンシングされる
- Block、Wedge、Ball、Cylinderなどのプリミティブは描画・物理ともに最も効率的
- 同じ形状・マテリアルのPartは自動バッチングされる
- 同じ形状でも使えるPart（直方体など）ならMeshPartより若干メモリ効率が良い（BlackoutCedarの検証：同一構造のPartモデルとMeshモデルをOBJ出力するとPart版の方がわずかにファイルサイズが小さい）

## 3-2. Union（CSG）

- 手軽に複雑な形状を作れるが、パフォーマンス面では注意が必要
- CSGアルゴリズムが不要なポリゴン（不可視面）を生成することがある
- シェーディングアーティファクトが発生しやすい
- 大量使用時にMeshPartより描画パフォーマンスが劣る傾向がある
- Union破損（消失バグ）のリスクがある
- RDC 2018で「Unionはローエンドデバイス向けに設計されていない」と言及されている
- インスタンシング自体は対応しているが、CSG内部データの保持によりメモリオーバーヘッドがMeshPartより大きい

**推奨:** Unionで形状を作った後、Export Selection → .OBJで書き出し → MeshPartとして再インポートすると、シェーディング問題の解消・破損リスクの排除・メモリ効率の改善が期待できる。MrChickenRocketの例では、100以上のPartからなる家をマテリアルごとにUnion → OBJ出力 → メッシュ化し、12ドローコール/100インスタンスから6ドローコール/6インスタンスに削減している。

## 3-3. MeshPart（Blender等で制作）

- 最もポリゴン効率が良い。頂点レベルで最適化可能
- シェーディングアーティファクトが（正しく作れば）発生しない
- 同じMeshId＋同じMaterialでバッチングが効く
- RenderFidelityによるLOD制御が利用可能
- SurfaceAppearanceでPBRテクスチャリングが可能
- 推奨される制作フロー: Blenderで低ポリモデリング → 不要面削除 → テクスチャベイク → glTF (.glb) / FBX / OBJ でエクスポート → Studioでインポート

# 4. メモリ負荷の最適化

## 確認方法と目標値

**何を見るか:** Stats詳細画面（Ctrl+Alt+F7）またはDeveloper Console（F9）で表示される **メモリ使用量**（ショートカットはスタジオ・クライアント共通）。ただしStudio内ではエディタのオーバーヘッドがあるため正確な値にならない。必ず実際のクライアント（公開テストプレイなど）で計測する。

**目標値（クライアント）:** クライアントメモリ 1.3GB以下（2GB デバイスではシステム全体の約 66% を超えると OS が強制終了するため）

**目標値（サーバー）:** サーバーメモリ使用率50%以下

**内訳の確認:** Developer Console の PlaceMemory カテゴリで、メモリを多く消費している上位の内訳（GraphicsTexture、PhysicsParts、Sounds など）を確認できる。上位に意図しないカテゴリがある場合は削減の優先ターゲットになる。

サーバー全体のメモリは以下の式で概算できる。

```
サーバー総メモリ ≈ 6.25 GiB + (100 MiB × 最大同時接続プレイヤー数)
```

例: 最大30人のサーバーの場合 → 約9.18 GiB

なお、サーバーはプレイヤーが接続するとメモリが増加するが、切断しても自動では減らない。サーバーが空になって再起動されたタイミングで初期値に戻る。

## 4-1. テクスチャサイズ

- テクスチャの解像度上限は **4096×4096（4K）**。Assets API 経由のアップロードは最大 8000×8000 まで受け付けるが、エンジン側の上限は 4K
- エンジンはエクスペリエンス読み込み時に自動的に低品質バージョンから始め、デバイスリソースに応じてクオリティを段階的に上げる（**テクスチャストリーミング**）。そのため高解像度テクスチャをアップロードしても、低スペックデバイスでは自動的にダウングレードされる
- ただし上限いっぱいのサイズを使うことは推奨されない。目安として、オブジェクトのサイズに合わせて以下を選ぶ：
  - 小物（5×5 studs 程度）: **256×256**
  - 中型（10×10 studs 程度）: **512×512**
  - 大型（20×20 studs 程度）: **1024×1024**
- グラフィックスメモリの消費量はディスク上のファイルサイズではなく**ピクセル数**で決まる。1024×1024 は 512×512 の **4倍**のグラフィックスメモリを消費するため、不必要に大きなテクスチャは使わない。1024px の SurfaceAppearance フルセット（ColorMap・NormalMap・RoughnessMap・MetalnessMap）で約 2.8MB、100 セット使うと約 300MB になる
- 同じ画像を別 ID で複数回アップロードすると重複分だけメモリを無駄に消費する。同一テクスチャは同じ asset ID を使い回す
- `MetalnessMap` を省略した場合のデフォルトは **0.0（非金属）**（公式ドキュメントに明記）。完全な非金属サーフェスはマップ自体を省略してよい。金属（1.0）や中間値の場合は 1x1 ピクセルのグレースケール画像を使う
- `RoughnessMap` のデフォルト値は公式に明記されていないため、単色の場合も 1x1 ピクセルで値を明示する
- 大きなテクスチャはヒーローアセット（武器、キャラクターなど近くで頻繁に見るもの）にのみ使用
- SurfaceAppearanceの数をできるだけ減らす。同じテクスチャセットを使うメッシュは結合する

## 4-2. Material Variant（マテリアルバリアント）の活用

`MaterialVariant` とは、Robloxのベースマテリアルをカスタマイズする仕組みである。`MaterialService` 内に `MaterialVariant` インスタンスを作成し、ColorMap・NormalMap・RoughnessMap・MetalnessMapとタイリング設定を定義することで、`BasePart.MaterialVariant` プロパティで任意のパートに適用できる。SurfaceAppearanceがMeshPart専用のPBRテクスチャであるのに対し、MaterialVariantは基本Partにも適用でき、タイル可能なテクスチャとして機能する。

**SurfaceAppearanceと比べた場合の利点：**
- **タイリング方式** — UV マッピングではなく studs 単位でテクスチャをタイルするため、小さなテクスチャで広い壁・床・地面をカバーできる。SurfaceAppearance は UV に縛られるため、広い面を高品質に見せるには高解像度テクスチャが必要になりがち
- **基本Partにも適用できる** — SurfaceAppearance は MeshPart 専用。床・壁・構造パーツなど基本Partで組まれたオブジェクトにもカスタムテクスチャを適用できる
- **バッチングとの相性** — 同じ MaterialVariant を使うパートは同じマテリアルとして扱われバッチが成立しやすい。SurfaceAppearance が異なると MeshPart 間のバッチが壊れる（セクション 2-1 参照）

なお、SurfaceAppearance でも同じ asset ID を複数の MeshPart で使い回せばテクスチャメモリは共有される。重要なのはアセットIDの使い回しであり、MaterialVariant に限った話ではない。

**実践：**
- ヒーローアセット（武器・近景のキャラクターなど、近くで見られるもの）には SurfaceAppearance を使い、背景・建物・地面など大量に並ぶオブジェクトには MaterialVariant を使う
- ヒーローアセット以外のテクスチャリングを MaterialVariant に寄せることで、ドローコールとメモリの両方を節約できる

## 4-3. 頂点カラー（Vertex Color）によるテクスチャレスな着色

テクスチャファイルを使わずに着色する手法として、Blender の **Vertex Paint** で各頂点に RGB 情報を埋め込む方法がある。テクスチャを持たないためグラフィックスメモリの消費がほぼゼロになる。

- ベースマテリアル（Plastic、Neon 等）・MaterialVariant・SurfaceAppearance と共存できる
- 頂点カラーはこれらのベースカラー（Part.Color・ColorMap）と**乗算**で合成される。Part.Color を白にしておくと頂点カラーがほぼそのまま表示される
- Blender で Vertex Paint を行い FBX または glTF (.glb) でエクスポートするとカラー情報が保持される
- **特に有効なケース:** 大量配置する植物・石・雑貨など、カラーバリエーションが必要だがテクスチャ品質がそこまで重要でないプロップ
- **制限:** 解像度は頂点密度に依存するため、細かいグラデーションや高精細なディテールには向かない

## 4-4. CollisionFidelity

Robloxのフィジクスエンジンは、ビジュアルメッシュとは**別に**コリジョン用のジオメトリデータを生成・保持する。CollisionFidelityの設定が複雑になるほど、このコリジョンデータが大きくなりメモリを消費する。

**Box** — 軸並行バウンディングボックス（6面）のみ。データ量が最小で最も軽量。プレイヤーが直接触れないオブジェクトや大まかな当たり判定で十分なものに使用

**Hull** — メッシュ全体を1つの凸包で包む。Boxより精度は高いが、凸包1つ分のデータが増える

**PreciseConvexDecomposition** — HACD アルゴリズムでメッシュを**複数の凸包**に分解する。形状が複雑なほど凸包の数が増え、それぞれの凸包データが積み重なるためメモリコストが高くなる。複雑な形状に正確なコリジョンが必要な場合のみ使用

**Default** — エンジン任せ。通常は Box に近い挙動

**推奨:** 装飾用プロップやプレイヤーが直接接触しないオブジェクトは Box、またはCanCollide / CanQuery / CanTouch をすべて false にしてコリジョン自体を無効にする。

## 4-5. インスタンス数

- 各インスタンス（Part, Model, Script等）はメモリを消費する。Part は 1 パーツあたり約 1KB が目安（100万パーツで約 1GB）
- 大量の小パーツで構成されたモデルは、可能であれば Blender 等にエクスポートしてメッシュを統合し、MeshPart として再インポートすることでインスタンス数を減らす
- Streaming Enabledを使用している場合でも、ストリーミング範囲内のインスタンス数がメモリに影響する

# 5. ネットワーク転送負荷の最適化

## 確認方法と目標値

**何を見るか:** Stats詳細画面（Ctrl+Alt+F7、スタジオ・クライアント共通）の **Recv** 欄。サーバーからクライアントへ毎フレーム送られるデータ量（KB/s）を示す。

**目標値:** Recv 40KB/s以下を目標。40KB/sを超えると物理オブジェクトの更新がスロットルされ始め、50KB/sで上限に達する。移動する物理アセンブリ40〜60個程度が安全圏。

## 5-1. 物理レプリケーション

レプリケーションとは、サーバー上のオブジェクトの状態（位置・回転など）を接続中のすべてのクライアントへ継続的に送信する仕組みである。オブジェクトが動くたびにその更新データが送信されるため、動くオブジェクトの数がネットワーク帯域に直結する。

- 移動する各物理アセンブリは約0.4〜0.9 KB/sのレプリケーション帯域を消費する
- コイン・ペットなど数百の動くオブジェクトを扱う場合は、マップ・プロップ設計だけでは対応しきれずゲーム設計レベルでの対策が必要になる

## 5-2. Anchoredの活用

- `Anchored = true` のオブジェクトは物理レプリケーション帯域を消費しない。マップのプロップ・建物・地形パーツ等、動かないオブジェクトはすべて Anchored にする
- スクリプトで動かす必要があるオブジェクトでも、動いていない間は Anchored にして帯域を節約できる

# 6. 物理CPU負荷の最適化

物理シミュレーションのCPU負荷は MicroProfiler の `physicsStepped` スコープで確認できる。絶対的な目標値はなく、`physicsStepped` が他のスコープに比べて突出して大きい場合や、フレームレートが目標値を下回る原因として疑われる場合に対策を検討する。

## 6-1. Adaptive Timestepping

`Workspace.PhysicsSteppingMethod` を **Adaptive**（デフォルト）に設定すると、物理エンジンがアセンブリを3つのソルバーアイランド（240 Hz / 120 Hz / 60 Hz）に自動分類し、**最大 2.5 倍の物理パフォーマンス改善**が得られる。

**Fixed（240 Hz）に変更すべきケース：**
- ピンボール・精密車両など高精度シミュレーションが必要な体験
- ほとんどのアセンブリが 240 Hz アイランドに収まる場合（Adaptive のオーバーヘッドが逆効果になる）

## 6-2. 装飾プロップの物理プロパティ

当たり判定・タッチ判定が不要なオブジェクトは3つのプロパティをすべて `false` にする。コリジョンジオメトリの保持も省略されるためメモリ節約にもなる。

| プロパティ | 役割 | 不要なら |
|---|---|---|
| `CanCollide` | 物理衝突 | `false` |
| `CanTouch` | Touched イベント発火 | `false` |
| `CanQuery` | レイキャスト・空間クエリ対象 | `false` |

3つすべて `false` にすると物理シミュレーション対象から除外され、CPU・メモリの両方で効果がある。

## 6-3. コンストレイント数の最小化

アセンブリ内のコンストレイント・ジョイント数が増えるほどソルバーの計算コストが増大する。可動ギミックを設計する際はコンストレイントを必要最小限に抑える。

## 6-4. 環境プロップに Humanoid を使わない

Humanoid インスタンスを環境プロップ（鳥・魚・動くギミックなど）に使用してはいけない。Humanoid は水泳・登り・ジャンプ・落下などの状態判定を毎フレーム実行するため、複数配置するだけで `physicsStepped` のコストが急増する。

環境生物や動くオブジェクトには Humanoid の代わりに **AnimationController** を使用する。AnimationController は Humanoid が持つ状態演算を一切持たず、アニメーション再生に必要な最小限の構成で動作するため、物理CPU負荷が大幅に低い。

# 7. インスタンスストリーミング

**インスタンスストリーミング**とは、Workspace 内のすべてのインスタンスを最初からクライアントに送るのではなく、**プレイヤーの現在地に近いインスタンスだけを動的に送信・削除する**仕組みである。遠くにあるオブジェクトはクライアントのメモリに存在しないため、描画負荷・メモリ消費・初回ロード時間をまとめて削減できる。そのためマップ規模を問わずインスタンスストリーミングの有効化を推奨する。（`Workspace.StreamingEnabled = true`）

## 確認方法

効果は専用の計測画面では直接確認できないが、以下の指標で間接的に把握できる。

- **Join サイズ / ロード時間** — 有効化により初回ロードのデータ量が減少（セクション10-6参照）
- **InstanceCount** — Stats詳細（Ctrl+Alt+F7）の `InstanceCount` がストリーミング範囲外のインスタンスを除外した数値になる
- **クライアントメモリ** — 範囲外オブジェクトがメモリに乗らないため使用量が減少（セクション10-5参照）

## 7-1. 基本動作

- プレイヤーの周囲にあるWorkspace内のインスタンスのみをクライアントに送信・保持する
- 離れたインスタンスはクライアントから削除される（`StreamOutBehavior` の設定による）

## 7-2. 主要プロパティ

**StreamingMinRadius** — この範囲内のインスタンスは常にストリーミングされる（最小保証範囲）

**StreamingTargetRadius** — メモリに余裕があれば、ここまで読み込もうとする目標範囲

**`StreamOutBehavior`**:

- Default — メモリ不足時のみストリームアウト
- LowMemory — Defaultと同じ
- Opportunistic — メモリに余裕があっても積極的にストリームアウト。大規模で詳細なワールドに推奨

**`ModelStreamingBehavior`** — `Model` コンテナ（インスタンスの外枠）の送信タイミングを制御する。

- **`Default`** — 参加時に Workspace 内のすべての `Model` コンテナを、中身に関係なく一括送信する。数万の `Model` が存在する大規模マップでは参加時の通信量が増大する
- **`Improved`** — `Model` コンテナは、その中の `BasePart` が実際にストリーム対象になるまでクライアントに送信されない。大規模マップでは `Improved` を推奨する

## 7-3. Model.StreamingMode

**ストリームイン/アウトの単位と挙動**を制御する。距離外での見た目の表現は `LevelOfDetail`（7-4）が担う。

- **Atomic** — モデル全体が一括でストリームイン/アウトされる。複雑なインタラクティブオブジェクトに有用
- **NonAtomic（Default）** — 子のBasePart単位でストリーミングされる
- **Persistent** — 常にクライアントに存在し、ストリームアウトされない。重要なゲームオブジェクトに使用
- **PersistentPerPlayer** — 特定プレイヤーに対してのみ永続化

## 7-4. Model.LevelOfDetail

**距離外で何を表示するか**（見た目の軽量表現）を制御する。ストリームイン/アウトの単位・挙動は `StreamingMode`（7-3）が担う。

| 値 | 挙動 |
|---|---|
| `Disabled` | 距離外では非表示 |
| `StreamingMesh` | 低精度の簡易インポスターとして表示 |
| `SLIM` | 距離に応じて段階的に解像度を下げた合成メッシュで表示。`StreamingMesh` より高品質 |
| `Automatic` | エンジンが自動選択（現在の挙動は `Disabled` と同じ）|

# 8. ロード時間の最適化

Join時間（初回ロード時間）はユーザー離脱率に直結する。ロードに 30〜45 秒かかるとプレイヤーは戻ってこなくなると言われている。長いロード時間の原因は主に「ダウンロードするアセットの量が多い」「不要なアセットがクライアントに送られている」「プリロードのやりすぎ」の3つに分類できる。

## 確認方法

**Join サイズの内訳を調べる:** Studio Settings（File > Studio Settings > Network）の **Print Join Size Breakdown** を有効化してゲームを開始すると、Output ウィンドウに**サイズ上位20インスタンス**とインスタンス種別ごとの割合が出力される。上位に来ているカテゴリが削減の優先ターゲットである（詳細はセクション10-6を参照）。

**ロード時間の簡易計測:** ストップウォッチによる目視で十分な精度が得られる。スクリプトによる計測方法はセクション10-7を参照のこと。

## 8-1. アセットの重複を排除する

Join時にダウンロードが必要なのは**ユニークなアセットID**の数だけである。同じ見た目のメッシュやテクスチャが別々のIDで登録されていると、その数だけ独立したダウンロードが発生する。

### マップを一括インポートしない

マップ全体を一括インポートするとJoin時間の悪化とインスタンシング（バッチング）の崩壊が同時に起きる。詳細はセクション2-1を参照のこと。

### 同じアセットは同じIDで使い回す

同じ画像・メッシュを別IDで複数回アップロードすると、重複した数だけダウンロードが走る。

- テクスチャは同一のアセットIDを使い回す。色違いは `SurfaceAppearance.Color` でティントを変えるだけでよい
- Packageを活用することで、同一アセットが同じIDを持つことを保証できる（セクション2-1参照）

### ベースマテリアルはダウンロード不要

Robloxのベースマテリアルはサーバーからのテクスチャダウンロードが発生しない。カスタムテクスチャが本当に必要な場所（ヒーローアセット）に絞り、背景・構造物にはベースマテリアルを使うことでダウンロードが必要なアセット数を減らせる。

なお MaterialVariant はカスタムテクスチャを使うためダウンロードは発生するが、タイリング方式のため小さいテクスチャサイズで広い面をカバーできる。SurfaceAppearance（UV方式）と比べてダウンロードサイズを抑えやすいという利点がある。

### Trim sheetsでユニークID数を削減

複数のテクスチャパターンを1枚のアトラスにまとめた Trim sheet を使うと、少数のアセットIDで豊富な見た目を実現でき、ダウンロードが必要なテクスチャ数が減少する（セクション4-1・セクション9-2参照）。

## 8-2. ストレージの使い分け

- **`ServerStorage`** — クライアントが不要なアセット・スクリプトはすべてここに置く
- **`ReplicatedStorage`** — Join 時にすべてクライアントへ送信される。実行時にクライアントが必要とするものだけに絞る
- **`ReplicatedFirst`** — 他のすべてより先に読み込まれる。ローディング画面など起動直後に必要なものだけに限定し、特に最小限を心がける

## 8-3. PreloadAsync の適切な使い方

`ContentProvider:PreloadAsync()` はアセットをバックグラウンドでダウンロードし、再生時のポップインを防ぐための機能である。しかし対象を広げすぎるとロード時間が大幅に悪化する。公式ドキュメント（Improve Performance）は Workspace 全体のプリロードを**明確な悪例**として挙げている。

**プリロードすべき対象:**

- ローディング画面で使う画像
- 重要なUIのボタン背景・アイコン
- スポーン地点周辺の重要アセット

**プリロードすべきでない対象:**

- Workspace全体
- 体験全域で使う音声をまとめてプリロード
- `ContentProvider.RequestQueueSize` を使って全アセットの完了を待つ方式（不安定かつロード時間が悪化する）

## 8-4. オーディオファイルのJoin負荷とメモリ

Robloxはインポート時に音声をトランスコードして内部形式で保存する。Join時にダウンロードされるのも**Roblox内部のトランスコード済みデータ**であり、元のファイル形式（MP3/OGG/WAV/FLAC）はダウンロードサイズに影響しない。

ダウンロードサイズ（Join負荷）とデコード後のメモリ消費は、どちらも**再生時間・チャンネル数・サンプルレート**によって決まる。BGMを複数配置しているゲームや長い音声アセットが多いゲームでは、Join時間とメモリの両面での増加に注意する。

**デコード後のメモリ消費の見積もり:**

音声はRAMにデコードされた状態（PCM）で保持されるため、ファイルサイズではなく以下の式で消費量が決まる（Roblox内部のビット深度は非公開のため概算）。

```
デコード後のメモリ（バイト） ≒ サンプルレート × 再生時間（秒） × チャンネル数 × (ビット深度 ÷ 8)
```

| 用途 | サンプルレート / 再生時間 / チャンネル / ビット深度 | 計算 | 概算メモリ |
| --- | --- | --- | --- |
| 短いSE | 44.1kHz / 3秒 / mono / 16-bit | 44,100 × 3 × 1 × 2 | 約 0.25 MB |
| 中尺SE | 44.1kHz / 10秒 / stereo / 16-bit | 44,100 × 10 × 2 × 2 | 約 1.7 MB |
| BGM（stereo） | 44.1kHz / 3分 / stereo / 16-bit | 44,100 × 180 × 2 × 2 | 約 30 MB |
| BGM（mono化） | 44.1kHz / 3分 / mono / 16-bit | 44,100 × 180 × 1 × 2 | 約 15 MB |

**Join サイズの確認方法:**

Studio の設定 → Network → **Print Join Size Breakdown** を有効にしてゲームを起動すると、Joinデータのサイズ上位20インスタンスが出力される。音声アセットが上位を占めている場合は削減を検討する。

**対策:**
- **BGMはmonoを検討する** — stereoはmonoの2倍のダウンロード・メモリを消費する。空間定位が不要な環境音・アンビエンスはmonoで十分
- **`PreloadAsync` に音声を含めすぎない** — 起動時のプリロード対象を絞ることでロード画面の待ち時間を短縮できる（セクション8-3参照）

## 8-5. Streaming Enabled との関係

`Workspace.StreamingEnabled = true` を有効にすると、Join時に送信されるデータをプレイヤー周辺のインスタンスに絞ることができ、初回ロード時間を大幅に削減できる。本セクションのすべての施策と組み合わせて使うことで相乗効果が得られる。詳細はセクション7を参照のこと。

# 9. 制作手法別の推奨事項まとめ

## 9-1. Studioでの Part制作

- Block, Wedge, Cylinder等のプリミティブを活用し、マテリアルの種類を統一してバッチ効率を最大化する
- 大量の小パーツで構成したモデルは、マテリアルごとにUnion → OBJ出力 → MeshPartとして再インポートすることでドローコールとインスタンス数を大幅に削減できる
- Unionのまま最終成果物にするのは非推奨（メモリオーバーヘッド、破損リスク、シェーディング問題）
- 装飾パーツは `Anchored = true`、`CanCollide = false`、`CastShadow = false`（必要に応じて）を設定
- 同じ形状のPartを大量配置する場合はバッチングが有効に働くため、Partのままでも効率的

## 9-2. Blender等での メッシュ制作

- ローポリを基本とし、ディテールはテクスチャで表現する
- 不可視面（Interior Faces）を必ず削除する
- 同じテクスチャセットを使うオブジェクトは結合して1メッシュにし、SurfaceAppearanceの数を減らす
- テクスチャは可能な限り512x512以下。単色のマップは1x1ピクセルに（`MetalnessMap` は非金属なら省略可。`RoughnessMap` はデフォルト不明のため 1x1 で明示）
- エクスポート形式: FBXなら1ファイルに最大200オブジェクト含められる。OBJはすべて1メッシュに結合される
- `CollisionFidelity` を適切に設定（装飾は `Box`、インタラクティブなものは必要に応じて `Hull` や `PreciseConvexDecomposition`）
- `RenderFidelity` は基本 `Performance`
- 再利用するプロップは同じMeshIdで配置し、色・サイズ・回転で変化をつけてバッチ効率を維持する

# 10. 負荷チェックの手順と計測画面の読み方

実際に計測を行う際の具体的な手順と、各画面の数値をどう読み解くかを説明する。

## 10-1. 計測の前提条件（ベンチマーク5ステップ）

最適化の効果を正しく評価するには、変更の前後で同じ条件で計測することが不可欠である。公式ドキュメント（Design for Performance）は低スペックモバイルを baseline として以下の5ステップを推奨している。

1. **基準端末（baseline）を1台固定する** — 低スペック端末でも快適に動作することがゲーム設計の最低ラインとなるため、低スペック寄りのモバイルを1台選んで固定し、以後の計測は常にその端末で行う。PC で 10ms の処理でも、1GHz 駆動の低スペックスマートフォンでは約 4 倍の 40ms（25fps）かかる場合があり、PC のみでの計測では問題を見落としやすい。端末が変わると数値の比較が意味をなさなくなるため、変更前後で必ず同じ端末を使う
2. **固定の計測ルートを用意する** — 街中の密集エリア→屋内→遠景オープンエリアなど、負荷が変化するルートを事前に決め、毎回同じルートを歩く
3. **確認する計測項目を固定する** — どのショートカットキーでどの統計を見るかを事前に決めておく（本セクションの10-3〜10-7を参照）
4. **グラフィックス品質を手動最大値に設定する** — クライアントの設定で Graphics Mode を Manual（手動）に切り替え、品質を最大値にする。自動調整が有効だと負荷が軽く見えてしまい、正確な計測ができない
5. **変更は1回に1要素、差分を記録する** — 複数の変更を同時に行うと何が効いたか分からなくなる。MicroProfilerのHTMLダンプ（9-3参照）を変更前後で保存しておくと定量的に比較できる

## 10-3. Rendering Stats の読み方

**開き方:** Studio: `Shift+F2` / クライアント: `Shift+F2`

Rendering Stats はリアルタイムで更新される描画統計画面である。ゲーム内を歩き回りながら、数値が大きく跳ね上がる場所（ホットスポット）を探す。

### Draw(Scene) — ドローコールとトライアングル数

`Draw(Scene)` 欄には2つの数値が表示される。

- 左の数値 = **ドローコール数**（エンジンがGPUに送った描画命令の回数）
- 右の数値 = **トライアングル数**（現在フレームに描画されているポリゴンの総数）

この2つは独立した指標である。ドローコールが少なくてもトライアングルが多すぎる場合や、その逆のケースもある。片方だけを見て「問題なし」と判断しないよう、常に両方を確認する。

**目標値（MrChickenRocket推奨）:**

- ドローコール数: 500以下
- トライアングル数: 500,000以下

### CPU ms と GPU ms — ボトルネックの判定

同じ画面に表示される `CPU ms` と `GPU ms` を比較することで、何がフレームレートのボトルネックになっているかを判定できる。

| 状態 | 意味 | 主な対策の方向性 |
| --- | --- | --- |
| **CPU ms > GPU ms** | CPUが描画のボトルネックになっている | ドローコール数の削減（インスタンシング改善）、スクリプト・物理の見直し |
| **GPU ms > CPU ms** | GPUが描画のボトルネックになっている | トライアングル数の削減、シャドウ・透明オブジェクトの削減 |

どちらか高い方がボトルネックである。両方が高い場合は重大な問題がある。

---

## 10-4. MicroProfiler の読み方

**開き方:** Studio / クライアント共通: `Ctrl+Alt+F6`

MicroProfilerはCPU負荷の詳細を1フレーム単位で可視化するツールである。スクリプト・物理・レプリケーションのどこに時間がかかっているかを特定できる。

### フレーム色の読み方

MicroProfilerの棒グラフはフレームごとに色分けされており、ボトルネックの種類をひと目で判断できる。

- **青** — Render優勢（描画負荷が支配的）
- **橙** — Jobs優勢（スクリプト・物理などのCPU負荷が支配的）
- **赤** — GPU Waitを含む（GPUが処理しきれず、CPUが待機している状態）

青や橙が続く場合はそれぞれ描画・CPU側の最適化が有効である。赤が頻出する場合はGPU負荷の深刻な過多を示しており、トライアングル数削減やライティング設定の見直しが優先される。

### HTMLダンプ保存による前後比較

MicroProfilerは `Ctrl+Alt+F6` でダンプ保存ダイアログを呼び出し、フレームの詳細データをHTMLファイルとして書き出せる。

**活用手順:**

1. 最適化を行う**前**にダンプを保存（ファイル名に「before」を付けると分かりやすい）
2. 最適化を実施する
3. 同じ計測シーン・同じ位置で再度ダンプを保存（「after」ファイル）
4. 2つのHTMLファイルをブラウザで開いて比較する

各処理ブロックの実行時間（ms）が数値で表示されるため、体感だけでなく定量的に効果を確認できる。

MicroProfiler には2つのダンプを直接 **Diff（差分比較）** する機能も追加されており、どのタグのコストが増減したかを一画面で把握できる。メモリプロファイリング・フレームグラフも同時に追加されている。

### Network ダンプ（verbosity: High）

MicroProfilerの Network ダンプは `verbosity` を `High` に設定することで、エンジンのレプリケーション・ストリーミングに加えて **HTTP asset delivery**（テクスチャ・メッシュ等のアセット配信）の内訳も確認できる。`verbosity: High` ではアセットIDも表示されるため、ロード中にどのアセットが配信遅延を引き起こしているかを特定できる。


---

## 10-5. Stats詳細 / Developer Console の読み方（メモリ・ネットワーク）

**開き方:** Stats詳細: `Ctrl+Alt+F7` / Developer Console: `F9`（Studio・クライアント共通）

### Studio内では計測しない

セクション4でも述べた通り、Studio内ではエディタのオーバーヘッドが上乗せされるため正確な値が得られない。メモリ使用量の計測は必ず実際のクライアント（公開テストプレイ、またはPlaytestのServer + 1 Client構成）で行う。

### リーク・膨張の兆候（Developer Console > Memory）

Developer Console の Memory タブで **LuaHeap** と **InstanceCount** を監視する。プレイ中に値が右肩上がりで増え続ける場合はリークを疑う。兆候の意味・原因の詳細はスクリプティングガイドを参照のこと。

### サーバーメモリの目標値と計算式

目標値と計算式の詳細はセクション4「確認方法と目標値」を参照のこと。

### Recv — ネットワーク転送負荷の確認

Stats詳細の `Recv` 欄には、サーバーからクライアントへ毎フレーム送られているデータ量（KB/s）が表示される。主に物理オブジェクトのレプリケーション量とRemoteEventのトラフィックが反映される。

**目標値:** `Recv` 40KB/s以下

40KB/sを超えると物理オブジェクトの更新がスロットルされ始め、50KB/sで上限に達する。マップ上に動く物理オブジェクトが多い場合や、RemoteEventを高頻度で送信しているスクリプトがある場合に数値が上昇する。対策の詳細はセクション5を参照のこと。


---

## 10-6. Join サイズの特定（Print Join Size Breakdown）

初回ロード時間が長い場合、どのインスタンス・種別がデータ量を押し上げているかを特定する機能である。

**手順:**

1. Studio の **File > Studio Settings > Network** タブを開く
2. **Print Join Size Breakdown** を有効にする
3. Studio でゲームを開始（Play または Play Here）する
4. Output ウィンドウに **サイズ上位20インスタンス** と**インスタンス種別ごとの割合**が出力される

出力結果で上位を占めるインスタンスが、Joinコスト削減の優先ターゲットである。たとえばMeshPartが上位に多ければ不要なメッシュの統合や削除を検討し、Scriptが多ければスクリプトのサイズや数を見直す。


---

## 10-7. ロード時間の簡易計測

ロード時間はストップウォッチによる目視計測で十分な精度が得られるが、`ReplicatedFirst` に以下のスクリプトを配置することで簡易的な数値計測も可能である。

```lua
local startTime = os.clock()

game.Loaded:Connect(function()
    local loadTime = os.clock() - startTime
    local roundedLoadTime = math.round(loadTime * 10000) / 10000
    print("Game loaded in " .. roundedLoadTime .. " seconds.")
    print("Number of instances loaded: " .. #workspace:GetDescendants())
end)
```

**注意点:**

- `game.Loaded` はクライアントのコンテンツ読み込み完了を示すイベントだが、HTTP asset deliveryの完了タイミングとは必ずしも一致しない。End-to-endの完全な計測ではないため、目安値として扱う
- ロード時間に影響する主な要因はインスタンス数・テクスチャサイズ・`ContentProvider:PreloadAsync()` の使用範囲である。`PreloadAsync` でWorkspace全体をプリロードするのは公式が明確に悪例として挙げており、ロード時間を大幅に悪化させる。必要なアセットだけを対象にする

# 12. 逆引きリファレンス — 症状から原因を探す

> 計測して問題を特定したが、何を直せばよいかわからない場合に使う。

---

### FPS が低い / フレームレートが不安定

| 疑うべき原因 | 確認箇所 |
|---|---|
| ドローコールが多すぎる | → 2-1（バッチング条件の確認）|
| トライアングルが多すぎる | → 2-2（ポリゴン数・不可視面）|
| `DoubleSided` が意図せず有効 | → 2-10 |
| 透明オブジェクトの重なり（Overdraw） | → 2-3 |
| VFX の同時存在数が多い | → 2-4 |
| `RenderFidelity` が `Precise` のまま | → 2-6 |
| ライティングが `Realistic` でシャドウキャスティングライトが多い | → 2-7 |
| `CastShadow` が広範囲に有効 | → 2-8 |
| Union（CSG）を多用している | → 3-2 |

---

### メモリ使用量が高い / クラッシュする

| 疑うべき原因 | 確認箇所 |
|---|---|
| テクスチャ解像度が高すぎる | → 4-1 |
| 単色マップが 1×1px になっていない | → 4-1 |
| SurfaceAppearance の数が多い | → 4-1 |
| テクスチャで着色しているプロップが多い | → 4-3 |
| Union の多用 | → 3-2 |
| インスタンス数が過多 | → 4-5 |
| 音声アセットが多い・長い | → 8-4 |

---

### ロード時間が長い / Join に時間がかかる

| 疑うべき原因 | 確認箇所 |
|---|---|
| マップを一括インポートした（ID重複） | → 2-1 / 8-1 |
| 同一アセットを複数IDでアップロード | → 8-1 |
| カスタムテクスチャを多用している | → 4-2 / 8-1 |
| `PreloadAsync` の対象が広すぎる | → 8-3 |
| ストレージの使い分けが適切でない | → 8-2 |
| 音声ファイルのサイズが大きい（stereo・長尺） | → 8-4 |
| Streaming Enabled が無効 | → 7 |
| `ModelStreamingBehavior` が `Default` のまま（大規模マップ） | → 7-2 |
| `Model.LevelOfDetail` が未設定（大型モデルが遠景でも重い） | → 7-4 |

---

### 帯域（Recv）が高い

| 疑うべき原因 | 確認箇所 |
|---|---|
| 動かないオブジェクトに `Anchored = false` | → 5-2 |
| 物理演算中のオブジェクトが多い | → 5-1 / 6-2 |

---

### CPU 負荷が高い（physicsStepped が大きい）

| 疑うべき原因 | 確認箇所 |
|---|---|
| `Workspace.PhysicsSteppingMethod` が `Adaptive` でない | → 6-1 |
| 物理シミュレーション対象から除外されていない装飾パーツが多い | → 6-2 |
| `CollisionFidelity` が `Default` のまま | → 4-4 |
| 環境プロップに `Humanoid` を使っている | → 6-4 |
| コンストレイント数が多い | → 6-3 |

---

### バッチングが効かない（ドローコールが下がらない）

| 疑うべき原因 | 確認箇所 |
|---|---|
| 同一メッシュを別IDで複数回インポートした | → 2-1 / 8-1 |
| `Transparency` に中間値が混入している | → 2-1 / 2-3 |
| マテリアル・テクスチャが統一されていない | → 2-1 |
| Package を使わずIDが分散している | → 8-1 |
| Union を最終成果物として使用している | → 3-2 / 9-1 |
